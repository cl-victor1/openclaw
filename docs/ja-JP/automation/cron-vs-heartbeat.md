# Cron vs Heartbeat: どちらを使うか

ハートビートとcronジョブはどちらもスケジュールに基づいてタスクを実行できます。このガイドは使用ケースに合った適切なメカニズムを選ぶのに役立ちます。

## クイック判断ガイド

| ユースケース | 推奨 | 理由 |
| ------------------------------------ | ------------------- | ---------------------------------------- |
| 30分ごとに受信箱を確認する | ハートビート | 他のチェックとまとめられ、コンテキスト認識 |
| 午前9時きっかりに日次レポートを送る | Cron（分離） | 正確なタイミングが必要 |
| カレンダーで近づくイベントを監視する | ハートビート | 定期的な認識に自然に適合 |
| 週次の詳細分析を実行する | Cron（分離） | スタンドアロンタスク、別のモデルを使用可能 |
| 20分後にリマインドする | Cron（メイン、`--at`） | 正確なタイミングを持つワンショット |
| バックグラウンドプロジェクトヘルスチェック | ハートビート | 既存のサイクルに便乗 |

## ハートビート: 定期的な認識

ハートビートは**メインセッション**で定期的な間隔（デフォルト: 30分）で実行されます。エージェントが物事を確認し、重要なことを表面化するために設計されています。

### ハートビートを使う場合

- **複数の定期チェック**: 受信箱、カレンダー、天気、通知、プロジェクトステータスを確認する5つの個別のcronジョブを作る代わりに、単一のハートビートでそれらすべてをまとめて処理できます。
- **コンテキスト認識の判断**: エージェントはメインセッションのフルコンテキストを持つため、何が緊急で何が待てるかについてスマートな判断ができます。
- **会話の連続性**: ハートビートの実行は同じセッションを共有するため、エージェントは最近の会話を覚えていて、自然にフォローアップできます。
- **低オーバーヘッドのモニタリング**: 1つのハートビートが多くの小さなポーリングタスクを置き換えます。

### ハートビートの利点

- **複数のチェックをまとめる**: 1回のエージェントターンで受信箱、カレンダー、通知を一緒に確認できます。
- **API呼び出しを削減**: 単一のハートビートは5つの分離されたcronジョブより安価です。
- **コンテキスト認識**: エージェントはあなたが取り組んでいることを知っており、それに応じて優先度を付けられます。
- **スマートな抑制**: 注意が必要なことが何もない場合、エージェントは `HEARTBEAT_OK` と返信し、メッセージは配信されません。
- **自然なタイミング**: キューの負荷に基づいてわずかにドリフトします。ほとんどのモニタリングには問題ありません。

### ハートビートの例: HEARTBEAT.mdチェックリスト

```md
# Heartbeat checklist

- Check email for urgent messages
- Review calendar for events in next 2 hours
- If a background task finished, summarize results
- If idle for 8+ hours, send a brief check-in
```

エージェントは各ハートビートでこれを読み、すべての項目を1回のターンで処理します。

### ハートビートの設定

```json5
{
  agents: {
    defaults: {
      heartbeat: {
        every: "30m", // インターバル
        target: "last", // 明示的なアラート配信ターゲット（デフォルトは "none"）
        activeHours: { start: "08:00", end: "22:00" }, // オプション
      },
    },
  },
}
```

完全な設定については [Heartbeat](/gateway/heartbeat) を参照してください。

## Cron: 正確なスケジューリング

cronジョブは正確な時刻に実行でき、メインコンテキストに影響を与えない分離セッションで実行できます。
繰り返しの毎時トップスケジュールは、0〜5分のウィンドウ内でジョブごとの決定論的なオフセットによって自動的に分散されます。

### cronを使う場合

- **正確なタイミングが必要**: 「毎週月曜日の午前9:00にこれを送る」（「午前9時頃」ではなく）。
- **スタンドアロンタスク**: 会話コンテキストを必要としないタスク。
- **異なるモデル/thinking**: より強力なモデルを使うべき重い分析。
- **ワンショットリマインダー**: `--at` を使った「20分後にリマインドして」。
- **ノイジー/頻繁なタスク**: メインセッション履歴を乱すようなタスク。
- **外部トリガー**: エージェントがそれ以外にアクティブかどうかに関わらず独立して実行されるべきタスク。

### cronの利点

- **正確なタイミング**: タイムゾーンサポート付きの5フィールドまたは6フィールド（秒）cron式。
- **組み込みの負荷分散**: 繰り返しの毎時トップスケジュールはデフォルトで最大5分までずらされます。
- **ジョブごとの制御**: `--stagger <duration>` でずらしをオーバーライド、または `--exact` で正確なタイミングを強制。
- **セッション分離**: メイン履歴を汚染せずに `cron:<jobId>` で実行。
- **モデルオーバーライド**: ジョブごとに安価または強力なモデルを使用。
- **デリバリー制御**: 分離ジョブはデフォルトで `announce`（サマリー）; 必要に応じて `none` を選択。
- **即時デリバリー**: Announceモードはハートビートを待たずに直接投稿。
- **エージェントコンテキスト不要**: メインセッションがアイドルまたはコンパクトされていても実行。
- **ワンショットサポート**: `--at` で正確な将来のタイムスタンプ。

### cronの例: 日次朝のブリーフィング

```bash
openclaw cron add \
  --name "Morning briefing" \
  --cron "0 7 * * *" \
  --tz "America/New_York" \
  --session isolated \
  --message "Generate today's briefing: weather, calendar, top emails, news summary." \
  --model opus \
  --announce \
  --channel whatsapp \
  --to "+15551234567"
```

これはニューヨーク時間の午前7:00ちょうどに実行され、品質のためにOpusを使用し、サマリーを直接WhatsAppにannounceします。

### cronの例: ワンショットリマインダー

```bash
openclaw cron add \
  --name "Meeting reminder" \
  --at "20m" \
  --session main \
  --system-event "Reminder: standup meeting starts in 10 minutes." \
  --wake now \
  --delete-after-run
```

完全なCLIリファレンスは [Cronジョブ](/automation/cron-jobs) を参照してください。

## 判断フローチャート

```
タスクを「正確な」時刻に実行する必要があるか？
  YES -> cronを使用
  NO  -> 続ける...

タスクをメインセッションから分離する必要があるか？
  YES -> cron（分離）を使用
  NO  -> 続ける...

このタスクを他の定期チェックとまとめることができるか？
  YES -> ハートビートを使用（HEARTBEAT.mdに追加）
  NO  -> cronを使用

これはワンショットリマインダーか？
  YES -> --atを使用してcronを使用
  NO  -> 続ける...

異なるモデルやthinkingレベルが必要か？
  YES -> cron（分離）を--model/--thinkingで使用
  NO  -> ハートビートを使用
```

## 両方を組み合わせる

最も効率的なセットアップは**両方**を使います:

1. **ハートビート**が30分ごとに1回のまとめられたターンで日常的なモニタリング（受信箱、カレンダー、通知）を処理します。
2. **Cron**が正確なスケジュール（日次レポート、週次レビュー）とワンショットリマインダーを処理します。

### 例: 効率的な自動化セットアップ

**HEARTBEAT.md**（30分ごとにチェック）:

```md
# Heartbeat checklist

- Scan inbox for urgent emails
- Check calendar for events in next 2h
- Review any pending tasks
- Light check-in if quiet for 8+ hours
```

**Cronジョブ**（正確なタイミング）:

```bash
# 午前7時に日次朝のブリーフィング
openclaw cron add --name "Morning brief" --cron "0 7 * * *" --session isolated --message "..." --announce

# 月曜日の午前9時に週次プロジェクトレビュー
openclaw cron add --name "Weekly review" --cron "0 9 * * 1" --session isolated --message "..." --model opus

# ワンショットリマインダー
openclaw cron add --name "Call back" --at "2h" --session main --system-event "Call back the client" --wake now
```

## Lobster: 承認付きの決定論的ワークフロー

Lobsterは、決定論的な実行と明示的な承認が必要な**マルチステップツールパイプライン**のワークフローランタイムです。
タスクが単一のエージェントターン以上で、人間のチェックポイントを持つ再開可能なワークフローが必要な場合に使用します。

### Lobsterが適する場合

- **マルチステップ自動化**: 1回限りのプロンプトではなく、ツール呼び出しの固定パイプラインが必要な場合。
- **承認ゲート**: 副作用は承認されるまで一時停止し、その後再開される必要がある場合。
- **再開可能な実行**: 以前のステップを再実行せずに一時停止したワークフローを継続する場合。

### ハートビートとcronとの組み合わせ方

- **ハートビート/cron**は実行が_いつ_起こるかを決定します。
- **Lobster**は実行が開始されたら_どのステップ_が起こるかを定義します。

スケジュールされたワークフローには、Lobsterを呼び出すエージェントターンをトリガーするためにcronまたはハートビートを使用します。
アドホックなワークフローには、直接Lobsterを呼び出します。

### 実装上の注記（コードより）

- Lobsterはツールモードで**ローカルサブプロセス**（`lobster` CLI）として実行され、**JSONエンベロープ**を返します。
- ツールが `needs_approval` を返した場合、`resumeToken` と `approve` フラグで再開します。
- ツールは**オプションのプラグイン**です。`tools.alsoAllow: ["lobster"]`（推奨）を通じて追加的に有効化します。
- Lobsterは `lobster` CLIが `PATH` で利用可能であることを期待します。

完全な使用方法と例については [Lobster](/tools/lobster) を参照してください。

## メインセッション vs 分離セッション

ハートビートとcronはどちらもメインセッションとやり取りできますが、方法が異なります:

|         | ハートビート | Cron（メイン） | Cron（分離） |
| ------- | ------------------------------- | ------------------------ | -------------------------- |
| セッション | メイン | メイン（システムイベント経由） | `cron:<jobId>` |
| 履歴 | 共有 | 共有 | 実行ごとに新規 |
| コンテキスト | フル | フル | なし（クリーンな状態で開始） |
| モデル | メインセッションモデル | メインセッションモデル | オーバーライド可能 |
| 出力 | `HEARTBEAT_OK` でなければ配信 | ハートビートプロンプト + イベント | Announceサマリー（デフォルト） |

### メインセッションcronを使う場合

以下が必要な場合は `--session main` と `--system-event` を使用してください:

- リマインダー/イベントをメインセッションコンテキストに表示させたい
- フルコンテキストで次のハートビート中にエージェントに処理させたい
- 別の分離実行を望まない

```bash
openclaw cron add \
  --name "Check project" \
  --every "4h" \
  --session main \
  --system-event "Time for a project health check" \
  --wake now
```

### 分離cronを使う場合

以下が必要な場合は `--session isolated` を使用してください:

- 前のコンテキストなしのクリーンなスレート
- 異なるモデルやthinking設定
- チャンネルへのAnnounceサマリー
- メインセッションを乱さない履歴

```bash
openclaw cron add \
  --name "Deep analysis" \
  --cron "0 6 * * 0" \
  --session isolated \
  --message "Weekly codebase analysis..." \
  --model opus \
  --thinking high \
  --announce
```

## コストの考慮事項

| メカニズム | コストプロファイル |
| --------------- | ------------------------------------------------------- |
| ハートビート | N分ごとに1ターン; HEARTBEAT.mdのサイズに比例してスケール |
| Cron（メイン） | 次のハートビートにイベントを追加（分離ターンなし） |
| Cron（分離） | ジョブごとにフルエージェントターン; 安価なモデルを使用可能 |

**ヒント**:

- トークンのオーバーヘッドを最小化するために `HEARTBEAT.md` を小さく保つ。
- 複数のcronジョブの代わりに類似のチェックをハートビートにまとめる。
- 内部処理のみが必要な場合はハートビートで `target: "none"` を使用する。
- 日常的なタスクには安価なモデルで分離cronを使用する。
